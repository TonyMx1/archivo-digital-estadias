Resumen: OCR para Buscador Inteligente (PDF/Word/Imagen)

Documento de referencia (consulta). Incluye opciones técnicas, flujo recomendado, uso de meta_doc, manejo de fotos desde cámara y precios aproximados de servicios de paga.
Objetivo

Permitir que, al subir un PDF/Word o capturar un documento con la cámara del celular, se extraiga el texto (OCR o parsing) y se almacene para que el buscador pueda encontrar documentos por contenido (ej. folio, oficio, nombres, números), no solo por el nombre del archivo.

1) Opciones de Implementación (Librerías / Servicios)

1.1 Librerías (sin costo por uso)

Herramienta Uso recomendado Notas
Tesseract.js OCR para imágenes (fotos) y PDFs escaneados (tras convertir páginas a imagen) Funciona en Node.js y navegador. Precisión depende de calidad de imagen.
pdf-parse Extraer texto de PDFs con texto embebido (no escaneados) Mucho más rápido que OCR cuando el PDF ya trae texto seleccionable.
mammoth Extraer texto de documentos Word .docx Bueno para obtener texto “crudo” sin depender de OCR.

1.2 Servicios en la nube (mayor precisión / escalabilidad)

Servicio Fortalezas Casos típicos
Google Cloud Vision API OCR general muy preciso, multi-idioma Documentos variados con mala calidad o necesidad de buena precisión.
AWS Textract Especializado en documentos; análisis de tablas y formularios Formatos estructurados (tablas, formularios, campos clave).
Azure Computer Vision (Read) Buena integración Microsoft; planes competitivos Organizaciones ya alineadas a Azure o con alto volumen.

2) Flujo Recomendado (Conceptual)

Usuario sube PDF/Word o captura imagen
  → Detectar tipo MIME
    → PDF: intentar pdf-parse
       → si no hay texto embebido: OCR (convertir a imagen por página)
    → Word: mammoth
    → Imagen (cámara): OCR directo con Tesseract
  → Guardar texto extraído (y metadatos) en BD
  → Buscador consulta por contenido (full-text recomendado)

3) Uso de la Columna meta_doc (metadatos)

Como ya existe meta_doc, una implementación práctica es guardar el contenido OCR dentro del JSON. Ejemplo:

{
  "texto_extraido": "...contenido extraído del documento...",
  "fecha_ocr": "2026-01-23T15:00:00Z",
  "metodo_extraccion": "pdf-parse" | "tesseract" | "mammoth",
  "paginas_procesadas": 5,
  "idioma": "es"
}
Ventajas de usar meta_doc
  • Evita cambios de esquema (sin columnas nuevas).
  • Flexible: puedes agregar campos nuevos en el futuro.
  • PostgreSQL permite consultar JSON y crear índices para acelerar búsquedas.

4) Caso Especial: Subida desde Cámara del Celular

Cuando el usuario toma una foto, el archivo llega como imagen (por ejemplo image/jpeg, image/png, image/webp).

En ese caso:

  • La foto no pasa por pdf-parse ni por mammoth.
  • Se aplica OCR con Tesseract.js directamente sobre la imagen.
  • Se almacena el texto extraído en meta_doc.texto_extraido para búsqueda.

Factores que afectan la calidad del OCR en fotos

Factor Impacto Recomendación
Iluminación Alto Buena luz, evitar sombras fuertes.
Enfoque / nitidez Alto Evitar movimiento; tomar foto nítida.
Ángulo Medio Capturar lo más perpendicular posible.
Resolución / compresión Medio No comprimir demasiado; conservar detalle del texto.

5) Búsqueda por Contenido (BD)

Para pocos documentos, una búsqueda simple puede funcionar. Para producción, es preferible búsqueda full-text (por ejemplo con PostgreSQL: to_tsvector + tsquery + índice GIN).
